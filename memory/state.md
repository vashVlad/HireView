# HireView — Current State

*Last updated: 2026-07-01 (seeded from HireView-Dev-Log.docx, Architecture-Redesign doc, and git log — update this in place as state changes, don't just append)*

## Status: live, in active use at Brillio

Deployed workflow tool for a recruiter working two roles at once at Brillio. Multi-machine by design (work + personal computer).

## What's shipped

- **JD Analyzer** — paste a JD, get a full LinkedIn Recruiter sourcing brief: wide (discovery, 5k–15k results) and narrow (execution, 500–1k) filter sets, boolean strings, seniority, industries, company size, spotlights.
- **Resume Screener** — bulk PDF/Word upload, Claude scores each against the JD (0–100, must-have 65% / nice-to-have 35% weighting), ranked results, history with status pipeline (New Applicant → Recruiter Screen → Contacted → Screening → Archived). Sub-45 scores excluded from history (raised from 30 on 2026-07-01).
- **Credibility Checker (Phase 4)** — optional second resume + LinkedIn PDF cross-referenced against the screened resume; surfaces discrepancies before a TA call. LinkedIn PDF stored to Supabase Storage during check. Replaced the old "compare" feature.
- **Projects layer** — a project = one open role. Supabase `projects` table; `screenings.project_id` FK. This became the app's core organizing unit.
- **Architecture Redesign (shipped 2026-06-29)** — nav collapsed from 5 items to 2: **Projects** and **All Candidates**. Old routes (`/jd-analyzer`, `/history`, `/tracker`, `/screener`) redirect. Every role/project is a self-contained workspace with tabs: **Filters** (LinkedIn filters, copy buttons), **Screen** (uploader scoped to the role), **Pipeline** (candidate cards — status, flags, notes, credibility), **Tracker** (swimlane TA→L1→L2→In-Person→Offer/Reject, drag-to-reorder, slide-over drawer), **Settings**.
- **Tracker tab (added 2026-06-30)** — was originally a separate page, now folded in as a 5th project tab. Excel export. Scheduled/unscheduled indicator dots.
- **Interview View (added 2026-07-06)** — two popup windows per candidate on pipeline cards: left-half resume/LinkedIn viewer (`/interview/[id]/document`) and top-right notes window (`/interview/[id]`) with AI-generated interview questions (cached, lazy) and debounced notes. Windows tile exactly using complement sizing math. "View resume" button in expanded card also opens the document window.
- **Card redesign (2026-07-06)** — filename removed, credibility signal emoji removed from collapsed state, career story shows role hierarchy (bold headers, subordinate bullets, softer summary), roles displayed oldest-first (chronological), Credibility Check section collapsible (default closed).

## Shipped, merged to main (2026-07-09/10, Phase 2.1 — Cross-Project Fit Suggestion) — final architecture, built by Claude Code this session on `feat/cross-project-fit-suggestion`

- **Cross-Project Fit Suggestion.** Per `HireView_Enterprise_Plan.md`'s Phase 2 — Intelligence Layer. On a below-threshold screening result (not yet saved to the pipeline), a "Check other active roles" link re-scores that resume against every other *active* project in the recruiter's own team, and surfaces the single best match if it clears that other project's own score threshold — e.g. FDE score 35 → suggests AI Developer at 73.
- **`app/api/cross-project-fit/route.ts`** — auth-gated, team-scoped via `getUserTeamIds()` (not `teamIdsFilter`, since this must always be team-bounded, even for admin — "same team" is the point of the feature). Calls the existing `scoreCandidate()` (imported, not edited) against each other active project's stored JD, 3 at a time (same concurrency cap as the primary screening route). `POST` response now carries the full scored `CandidateResult` and the target project's `jobDescription` alongside `{projectId, projectName, score}`, so a follow-up "Transfer" action can save directly without re-scoring.
- **Auto-fire rebuilt a fourth time, this time on a cheap Claude classification call — no local heuristic left standing.** Three consecutive local/free gates (must-have score, overall score, keyword overlap against other projects' must-haves) were each checked against a real failing test case (Alex Kim / AI Developer, mustHaveScore 32 → 82 elsewhere) and all three missed it. Root cause traced precisely for the keyword-overlap version: it matched against `result.careerTrajectory`, and `scoreCandidate`'s tool schema (`lib/scoreCandidate.ts`) explicitly generates that field scoped to "the role being hired for" — the CURRENT project. So the text being keyword-matched was never neutral; it's already filtered through the current role's lens and can omit a different role's relevant vocabulary regardless of résumé/JD length. Confirmed: only ~1/5 of the AI Developer JD's must-have vocabulary appeared anywhere in Alex Kim's current-role-scoped summary. This isn't a threshold problem — it's evidence that no cheap local heuristic reliably substitutes for actual semantic judgment.
- **New `POST /api/cross-project-fit/gate`** — re-extracts the real resume text (same `extractResumeText` the full check already uses, not the role-scoped summary) and makes one cheap Claude call with a minimal output schema (`{promising: boolean}`) against short role-name + must-have-list summaries (not full JDs) for every other active project. Real judgment, not free, but far cheaper than the full check's N complete scoring passes. Fails closed (`promising: false`) on any error.
- **`ResultCard.tsx`** — reverted to a simple `otherActiveCount: number` prop (the client-side skill data from the keyword-overlap attempt is no longer needed, now that the gate decision happens server-side); new `onCheckCrossProjectPromise?: () => Promise<boolean>` prop, called once via a guarded `useEffect` (states: `checkingGate`, `gateChecked`), chains straight into the existing `handleFindBetterFit()` when the gate says yes. UI shows the same "Checking other active roles…" text through both the gate phase and the real-check phase — no visible seam between them.
- **`app/projects/[id]/page.tsx`** — `onCheckCrossProjectPromise` routed through the same `fitQueueRef` single-flight queue as the real check, so gate calls and real checks never run concurrently either.
- **Explicitly not guaranteed** — Vlad asked directly; told him no classifier is 100%, but this doesn't share the specific structural blind spot (role-scoped summary matching) that killed the prior three attempts.
- **A real bug was found and fixed during this build:** `ResultCard.tsx`'s auto-fire `useEffect` listed `checkingGate` (state it sets inside itself) in its own dependency array, causing a self-cancelling loop that left "Checking other active roles…" stuck forever. Fixed with a `useRef`-based re-entry guard instead. See [[feedback_useeffect_self_cancel]] in the global memory vault — saved as a durable cross-session lesson.
- **Live-tested by Vlad after the fix — confirmed working.** The gate resolved correctly (not stuck) and correctly identified a real cross-project fit that all three prior gate mechanisms had missed (the Alex Kim / AI Developer case).
