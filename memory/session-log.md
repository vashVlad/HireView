Fixed by replacing the state-based re-entry guard with a `gateStartedRef` (a `useRef`, which doesn't trigger re-renders or effect re-runs) and removing `checkingGate`/`gateChecked`/`fitChecked`/`checkingFit` from the dependency array entirely ‚Äî none of them are real *inputs* to when this effect should start, they're outputs it produces. **Lesson for this codebase: never list state that an effect sets synchronously within its own body as a dependency of that same effect ‚Äî it can self-cancel mid-flight.** Same pattern existed in the prior keyword-overlap version's effect too (`fitChecked`/`checkingFit` were both deps and were-being-set-via `handleFindBetterFit` called from within), just never triggered because that gate's `isPromising` was false on the one test case exercised (Alex Kim) ‚Äî a latent bug that would likely have surfaced on any candidate the keyword gate actually did fire for.
