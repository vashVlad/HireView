**Update — Vlad reported "Checking other active roles…" stuck permanently, never resolving.** Real bug in `ResultCard.tsx`'s new auto-fire `useEffect`, not a network/data issue — confirmed the resume-scan-timing theory Vlad first suggested was wrong (re-read `screen-resumes/route.ts`: extraction and scoring are both fully awaited before the API responds, so the résumé is never incomplete when the card first renders), which narrowed it to the gate effect itself. Root cause: `checkingGate` was both set inside the effect AND listed in the effect's own dependency array. `setCheckingGate(true)` triggers a re-render → React sees the dependency changed → tears down that effect instance (flips its `cancelled` closure variable to `true`) → starts a second instance that correctly no-ops. But the original async call is still in flight; when it resolves, it checks `cancelled`, sees `true`, and exits *before* ever calling `setCheckingGate(false)` — permanently stuck. Fixed by replacing the state-based re-entry guard with a `gateStartedRef` (a `useRef`, which doesn't trigger re-renders or effect re-runs) and removing `checkingGate`/`gateChecked`/`fitChecked`/`checkingFit` from the dependency array entirely — none of them are real *inputs* to when this effect should start, they're outputs it produces. **Lesson for this codebase: never list state that an effect sets synchronously within its own body as a dependency of that same effect — it can self-cancel mid-flight.** Same pattern existed in the prior keyword-overlap version's effect too (`fitChecked`/`checkingFit` were both deps and were-being-set-via `handleFindBetterFit` called from within), just never triggered because that gate's `isPromising` was false on the one test case exercised (Alex Kim) — a latent bug that would likely have surfaced on any candidate the keyword gate actually did fire for.
