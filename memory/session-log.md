## 2026-07-20 (newest of all, round 25) — Real bug: a plain second resume with a LinkedIn URL in its header got misdetected as a LinkedIn PDF export

Vlad: "cross-reference got confused and it took a second resume as a linkedin profile... LinkedIn icon and activity shows when the pdf from LinkedIn is uploaded, but it doesn't have to show when a second resume is being uploaded."

- Root cause: `lib/assessCredibility.ts`'s `detectLinkedIn()` was a plain OR of three signals — a LinkedIn profile URL, a connections count, or a "Skills & Endorsements" heading. The URL signal alone is nearly always true for ordinary resumes too, since almost every modern resume lists the candidate's own LinkedIn URL as a contact detail in the header — so any second-resume cross-reference with a LinkedIn link in its header tripped this, showing the LinkedIn icon and the LinkedIn-signals activity panel (`CredibilitySection.tsx`'s `isLinkedIn = !!assessment.linkedInSignals`) for something that wasn't an actual export.
- Fixed: now requires at least 2 of the 3 signals together. The connections count and "Skills & Endorsements" heading are both wording specific to LinkedIn's own PDF export format — a genuine export reliably has one of those alongside the URL, while a plain resume with just a LinkedIn link in its header only ever satisfies 1 of 3 and no longer trips it.
- Single point of definition/usage (`app/api/assess-credibility/route.ts`'s only call), so this fix covers every call site of the credibility checker automatically — Pipeline, All Candidates, ResultCard, and the name-match compare flow.
- `npx tsc --noEmit -p tsconfig.json`: clean.
- Not yet committed — same uncommitted batch as everything else this session.

## 2026-07-20 (newest of all, round 24) — Credibility check could hang forever with no error: added a client-side timeout

Vlad: ran a cross-reference check on a Pipeline-tab result card, "took so long to process and gave no result." Traced through `components/CredibilityChecker.tsx`'s `runCheck()` — it had no timeout of its own on the `/api/assess-credibility` fetch at all, and that route (`app/api/assess-credibility/route.ts`, not do-not-touch) has `maxDuration = 60`, same class of ceiling as the screening batch timeout diagnosed earlier this session — except here there's only ever one Claude call (comparing the full resume against the full cross-reference doc), so a single unusually large document pair can genuinely push close to that limit. When Vercel kills a function past its ceiling, the connection doesn't always tear down cleanly enough for `fetch` to reject — worst case, the promise never settles at all, leaving "Checking…" spinning forever with zero feedback and no way to retry short of leaving the page.

- `components/CredibilityChecker.tsx` — `runCheck()` now uses an `AbortController` with a 55-second timeout (just under the route's own 60s ceiling), so this always resolves to a clear, actionable error ("Credibility check timed out — this can happen with an unusually long resume or cross-reference document. Try again, or with a smaller file.") instead of an indefinite silent hang. Distinguishes a real timeout (`DOMException` named `AbortError`) from any other error so the message stays accurate.
- Not Pipeline-specific — this same fix covers every call site (Pipeline, All Candidates, ResultCard, AlreadyScreenedCard's future use if any) since they all go through this one shared component.
- `npx tsc --noEmit -p tsconfig.json`: clean.

## 2026-07-15 (same day, follow-on #19) — Fixed popover/card z-index overlap bug at the CSS root

- Vlad flagged a screenshot: the Team page's "Add member" popover for one team was visually bleeding into/overlapping the next team card below it — avatars, names, and dropdown list all jumbled together.
- Root cause: `app/globals.css`'s `@keyframes fade-in-up` ended on `transform: translateY(0)`, not the literal keyword `none`. Any non-`none` computed transform — even a visually-identity one — makes the element a new CSS stacking context, and `animation-fill-mode: both` holds that computed value forever after the animation finishes. Every animated `<li>` in a list (team cards, candidate cards, etc.) was therefore its own permanent stacking context, so a later sibling's own contents could paint over an earlier sibling's popover regardless of the popover's own `z-50` — z-index only resolves within a shared stacking context, not across separate sibling ones. Same bug class as the click-outside overlay fix from earlier this session (task #180), different symptom.
- Fixed at the root: changed the keyframe's end state to `transform: none;` (visually identical — `translateY(0)` was already a no-op offset — but exempt from the stacking-context rule). This is a one-line, general fix that covers every current and future use of `.animate-fade-in-up` (confirmed still active in `ResultCard.tsx` and `ResumeUploader.tsx` in addition to the Team page), so it doesn't need to be reapplied per-component.

## 2026-07-17 (later still, same session) — Real bug found + fixed: silent resume_content_hash write failure

Vlad: "why existing resumes get screened again? Where's the warning?" Follow-up narrowed it down: same project, identical file, zero warning at all — but the candidate showed up flagged "Previously seen" instead of the expected "already screened, skipped scoring" card, meaning it went through full re-scoring rather than being caught by the free pre-check.

- **Root cause, found by reading `lib/screenings.ts`'s `saveScreening()`:** the `resume_content_hash` write (a cheap, deterministic operation — just hash the extracted text) lived inside the SAME `try` block as `generateFingerprint()`, which makes a real Claude tool-use call. Any failure in that call — rate limit, timeout, malformed tool response — throws, and the shared `catch` (documented as "best-effort, must never block the save") swallows it with only a server-side `console.error` Vlad would never see. That silently skipped the hash write too, even though the hash has nothing to do with fingerprinting.
- **Consequence:** once a screening's `resume_content_hash` never gets written, it can NEVER be caught by the exact-match pre-check (`app/api/screen-resumes/check-existing`) again — a later re-upload of the byte-identical file reads as "new," sails past the free duplicate check, and gets fully re-scored (burning an unnecessary Claude call). The only thing that might still catch it afterward is the separate, weaker fingerprint-similarity match (Phase 1.1/1.4) — which explains why the user saw a post-score "Previously seen" cross-project banner instead of the pre-score `AlreadyScreenedCard` warning that should have skipped scoring entirely: the original screening's own fingerprint may have also been a casualty of the same failed try block, while a different (cross-project) copy's fingerprint happened to succeed and matched instead.
- **Fix:** `resumeText` extraction + the `resume_content_hash` write now get their own dedicated `try/catch`, running BEFORE and independent of the fingerprinting `try` block. A fingerprinting failure can no longer take the hash write down with it. The second (fingerprinting) block reuses the already-extracted `resumeText` if available, or re-extracts if the first block itself failed.
- **This does not fix or retroactively backfill any already-affected rows** — any screening saved before this fix, on an occasion where fingerprinting happened to fail, still has `resume_content_hash: null` and will never self-heal. Worth a one-time backfill pass if this turns out to be common (would need to re-extract text for every hash-null row and hash it) — not attempted this session, flagging as a possible follow-up.
- **Verified via direct `Read` only** — this sandbox has no network access to Supabase (confirmed this session: `curl` to the Supabase REST API is blocked by the sandbox's outbound proxy allowlist), so the actual rate of NULL `resume_content_hash` rows in the live DB could not be checked directly. A real count (e.g. `SELECT count(*) FROM screenings WHERE resume_content_hash IS NULL`) would show how many candidates are currently exposed to this gap — worth running from Vlad's own machine or via Claude Code.
